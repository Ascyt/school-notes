# Maven und Gradle als Build Umgebung

## Funktion: Wofür ist die Build Umgebung zuständig?
Eine **Build Umgebung** sorgt dafür, dass aus dem Quellcode eine installierbare Android-App wird. Typische Aufgaben sind:

- Kotlin- und Java-Code kompilieren
- Ressourcen verarbeiten (Bilder, Strings, Themes)
- AndroidManifest zusammenführen (Manifest-Merge)
- App-Paket erzeugen (Android App Bundle oder Android Application Package)
- Signieren (Debug und Release)
- Tests ausführen (Unit-Tests, Instrumentation-Tests)
- Qualitätschecks ausführen (Lint, statische Analysen)
- Bibliotheken einbinden (Dependencies)

In Android übernimmt das praktisch immer **Gradle** (mit Android Gradle Plugin). **Maven** ist trotzdem wichtig, weil viele Bibliotheken über Maven-Repositories verteilt werden.

---

## Konfiguration: minSdk, targetSdk, compileSdk und maxSdk

### minSdk (MinSDK)
- Legt fest, **ab welcher Android-Version** die App überhaupt installiert und gestartet werden darf.
- Je höher die minSdk, desto weniger alte Geräte werden unterstützt.
- Je niedriger die minSdk, desto mehr Kompatibilität muss beachtet werden.

### targetSdk (Ziel-SDK)
- Legt fest, für **welche Android-Version** die App "ausgelegt" ist.
- Wichtig, weil Android bei neuen Versionen oft **Verhaltensänderungen** einführt (zum Beispiel bei Berechtigungen oder Hintergrunddiensten).
- Eine hohe targetSdk ist wichtig, damit die App auf aktuellen Android-Versionen korrekt und sicher läuft.

### compileSdk
- Legt fest, **gegen welche Android-API** das Projekt kompiliert.
- Bestimmt, welche neuen Klassen und Funktionen beim Programmieren verfügbar sind.
- compileSdk kann höher sein als minSdk, weil neue APIs oft nur zur Compile-Zeit gebraucht werden, die Laufzeit wird über Checks abgesichert.

### maxSdk (MaxSDK)
- In Android-Projekten wird ein "MaxSDK" **normalerweise nicht** als allgemeine Projektgrenze gesetzt.
- Es gibt zwar `maxSdkVersion` in bestimmten Manifest-Bereichen (zum Beispiel bei einzelnen Berechtigungen), aber das wird selten empfohlen, weil es Updates und Kompatibilität unnötig blockieren kann.

**Wo wird das konfiguriert?**  
Diese Werte werden typischerweise in den Gradle-Projektdateien im Android-Block der Modul-Konfiguration gepflegt (meist im App-Modul).

**Beispiel (Kotlin DSL, `build.gradle.kts` im App-Modul):**

```kotlin
android {
  compileSdk = 34

  defaultConfig {
    applicationId = "com.example.app"
    minSdk = 24
    targetSdk = 34
    versionCode = 1
    versionName = "1.0"
  }
}
```

---

## Maven
- Maven ist ein Build-Tool mit einem zentralen Projektmodell (**pom.xml**) und klaren Standard-Konventionen.
- Für Android-Apps wird Maven heute kaum als Build-System verwendet.
- **Wichtig ist Maven trotzdem**, weil viele Android-Libraries in Maven-kompatiblen Repositories veröffentlicht werden.

### Dependencies in Maven
Libraries werden über eindeutige Koordinaten beschrieben:
- group
- artifact
- version

**Beispiel (Ausschnitt aus `pom.xml`):**

```xml
<dependencies>
  <dependency>
    <groupId>com.squareup.okhttp3</groupId>
    <artifactId>okhttp</artifactId>
    <version>4.12.0</version>
  </dependency>
</dependencies>
```

Diese Denkweise nutzt Gradle im Android-Ökosystem ebenfalls.

---

## Gradle
- Gradle ist das Standard-Build-System für Android.
- Mit dem **Android Gradle Plugin** werden Android-spezifische Build-Schritte möglich:
  - Ressourcenverarbeitung, Manifest-Merge, Paket-Erstellung, Signierung, Build-Varianten (Debug und Release)

### Dependencies in Gradle
- Gradle lädt Libraries aus Repositories (zum Beispiel Google und Maven Central) und bindet sie in den Build ein.
- In größeren Projekten sind **Version Catalogs** üblich:
  - zentrale Datei `libs.versions.toml`
  - Versionen und Libraries werden an einer Stelle gepflegt, Module referenzieren nur noch die Einträge

**Beispiel (Kotlin DSL, `build.gradle.kts`):**

```kotlin
repositories {
  google()
  mavenCentral()
}

dependencies {
  implementation("androidx.core:core-ktx:1.13.1")
  implementation("androidx.compose.ui:ui:1.6.8")
}
```

**Beispiel (Version Catalogs):**

`gradle/libs.versions.toml`

```toml
[versions]
compose = "1.6.8"

[libraries]
compose-ui = { module = "androidx.compose.ui:ui", version.ref = "compose" }
core-ktx = { module = "androidx.core:core-ktx", version = "1.13.1" }
```

`build.gradle.kts`

```kotlin
dependencies {
  implementation(libs.core.ktx)
  implementation(libs.compose.ui)
}
```

### Wrapper
- Der **Gradle Wrapper** stellt sicher, dass überall dieselbe Gradle-Version verwendet wird.
- Dadurch laufen Builds reproduzierbar auf allen Rechnern und in Continuous Integration-Systemen.

**Beispiel (typische Wrapper-Dateien / Aufruf):**

- Im Projekt liegen Skripte wie `gradlew` / `gradlew.bat`.
- Build (Windows):

```bat
gradlew.bat assembleDebug
```

---

## Kurzvergleich

| Kriterium | Maven | Gradle |
|---|---|---|
| Rolle im Android-App-Build | selten | Standard |
| Stärken | klare Standards, einfache Struktur | flexibel, Plugins, Multi-Modul, Build-Varianten |
| Dependencies | Koordinaten (group, artifact, version) | Koordinaten + zentrale Verwaltung (Version Catalogs möglich) |
| Typischer Nutzen in Android | Repository- und Library-Welt | komplettes Build-System der App |

# Composables, Modifiers, Layout

## Grundidee: Benutzeroberfläche wird deklarativ beschrieben
Jetpack Compose arbeitet **deklarativ**. Das bedeutet: Es wird nicht Schritt für Schritt beschrieben, *wie* einzelne Elemente gezeichnet werden, sondern *wie die Oberfläche aussehen soll*, abhängig vom aktuellen Zustand.

- Der aktuelle **Zustand** bestimmt, was sichtbar ist (zum Beispiel Text, Farben, Listeinträge).
- Wenn sich der Zustand ändert, wird die betroffene Oberfläche **neu aufgebaut** (Recomposition).
- Dadurch entsteht eine Oberfläche, die sich automatisch anpasst, ohne dass an vielen Stellen manuell aktualisiert werden muss.

---

## Composables
**Composables** sind die kleinsten Bausteine in Jetpack Compose. Ein Composable beschreibt einen Teil der Oberfläche, zum Beispiel:

- ein Button
- eine Textzeile
- eine Karte
- ein kompletter Bildschirm (Screen)

**Beispiel (ein einfaches Composable):**

```kotlin
  @Composable
  fun Greeting(name: String) {
    Text(text = "Hallo, $name")
  }
```

### Wie Composables beim Aufbau helfen
Beim Aufbau einer grafischen Benutzeroberfläche wird die Oberfläche in kleinere Einheiten zerlegt:

- **Screen-Composable**: bildet die gesamte Seite ab (zum Beispiel "Login", "Liste", "Details")
- **Section-Composable**: größere Bereiche innerhalb eines Screens (zum Beispiel Kopfbereich, Inhalt, Fußbereich)
- **Element-Composable**: einzelne Komponenten (zum Beispiel Eingabefeld, Button, Eintrag in einer Liste)

Diese Zerlegung ist wichtig, weil dadurch:
- Änderungen leichter werden (nur ein Baustein wird angepasst)
- Bausteine wiederverwendbar sind (ein Listeneintrag kann in mehreren Screens genutzt werden)
- das Layout übersichtlicher bleibt

### Stateless und Stateful
Für saubere Oberflächen ist eine klare Trennung hilfreich:

- **Zustandslose Composables** (stateless):  
  bekommen alle Daten über Parameter und geben nur Oberfläche zurück.  
  Vorteil: leicht testbar, leicht wiederverwendbar, sehr klar.

- **Zustandsbehaftete Composables** (stateful):  
  verwalten eigenen Zustand (zum Beispiel Eingabetext).  
  Vorteil: schnell für kleine Teile, aber sollte kontrolliert eingesetzt werden.

In größeren Anwendungen wird Zustand oft nicht tief in der Oberfläche versteckt, sondern zentral verwaltet (zum Beispiel in einem ViewModel). Dadurch bleibt die Oberfläche stabil und nachvollziehbar.

---

## Layout
Ein **Layout** beschreibt, wie Elemente angeordnet werden. Folgendes wird dafür von Compose zur Verfügung gestellt:

- **Column**: Elemente vertikal untereinander (typisch für Formulare oder Seitenaufbau)
- **Row**: Elemente horizontal nebeneinander (typisch für Symbol + Text oder Button-Leiste)
- **Box**: Elemente übereinander stapeln (zum Beispiel Bild mit Text darüber, Badge, Overlay)

**Beispiel (Column/Row/Box):**

```kotlin
  @Composable
  fun ProfileHeader(userName: String) {
    Row(
      modifier = Modifier.fillMaxWidth().padding(16.dp),
      horizontalArrangement = Arrangement.SpaceBetween
    ) {
      Column {
        Text(text = "Willkommen")
        Text(text = userName)
      }

      Box {
        Icon(imageVector = Icons.Default.Person, contentDescription = null)
      }
    }
  }
```

### Typische Layout-Ziele in einer App
Beim Aufbau einer grafischen Benutzeroberfläche sind meist diese Ziele relevant:

- **klare Struktur** (Kopfbereich, Inhalt, Navigation)
- **gute Abstände** (Lesbarkeit und Touch-Bedienung)
- **Ausrichtung** (zentriert, linksbündig, gleichmäßig verteilt)
- **responsives Verhalten** (passt sich an Bildschirmgröße an)

### Listen und große Datenmengen
Sobald viele Elemente angezeigt werden (zum Beispiel eine Blog-Liste), werden sogenannte **Lazy**-Layouts verwendet (zum Beispiel LazyColumn).  
Dabei werden nur die Elemente aufgebaut, die wirklich sichtbar sind. Das verbessert die Performance und spart Speicher.

**Beispiel (LazyColumn):**

```kotlin
  @Composable
  fun SimpleList(items: List<String>) {
    LazyColumn {
      items(items) { item ->
        Text(
          text = item,
          modifier = Modifier.fillMaxWidth().padding(12.dp)
        )
      }
    }
  }
```

### Standard-Struktur für Apps
Viele Oberflächen folgen einer wiederkehrenden Struktur:
- oben eine Leiste (Titel, Aktionen)
- in der Mitte Inhalt (Listen, Formulare)
- unten Navigation oder Aktionsbereich

---

## Modifiers
Ein **Modifier** ist in Compose das wichtigste Werkzeug, um ein Element zu verändern. Modifiers werden genutzt für:

- **Größe** (Breite, Höhe)
- **Abstand** (Padding, Margin-ähnliche Abstände)
- **Ausrichtung** (Alignment innerhalb von Layouts)
- **Hintergrund und Form** (Farbe, Rundungen)
- **Interaktion** (Klick, Scrollen, Gesten)
- **Zugänglichkeit und Tests** (Semantik, Test-Markierungen)

### Warum Modifiers wichtig sind
Modifiers machen die Oberfläche flexibel, ohne die eigentliche Komponente umzuschreiben:

- Das Composable beschreibt "was es ist" (zum Beispiel Button, Text, Karte)
- Der Modifier beschreibt "wie es sich verhält und aussieht" (zum Beispiel Abstand, Klickfläche, Ausrichtung)

### Reihenfolge ist wichtig
Modifiers werden "von außen nach innen" angewendet. Dadurch kann die Reihenfolge das Ergebnis verändern, zum Beispiel:

- zuerst Hintergrund, dann Padding wirkt anders als zuerst Padding, dann Hintergrund
- Klickfläche kann größer oder kleiner werden, je nachdem, wann Größe und Padding gesetzt werden

Dieser Punkt ist beim GUI-Aufbau besonders wichtig, weil falsche Reihenfolge zu unerwartetem Aussehen oder schlechter Bedienbarkeit führen kann.

**Beispiel (Modifier-Reihenfolge):**

```kotlin
// Padding "innen" (Hintergrund umfasst auch Padding)
Modifier.background(Color.Gray).padding(16.dp)

// Padding "außen" (Hintergrund nur um den Inhalt)
Modifier.padding(16.dp).background(Color.Gray)
```

---

## Ergebnis: Wie daraus eine grafische Benutzeroberfläche entsteht
Der Aufbau folgt in Compose meistens diesem Muster:

1. Ein Screen wird als Composable definiert.
2. Der Screen wird mit Layouts strukturiert (Column, Row, Box, Listen).
3. Jedes Element erhält Modifiers für Aussehen und Bedienung.
4. Daten kommen über Parameter oder aus einem Zustand, der außerhalb der Oberfläche verwaltet wird.

Damit entsteht eine Oberfläche, die klar strukturiert ist und sich stabil erweitern lässt.


# Events und State

## Events
**Events** sind Benutzeraktionen oder Systemereignisse, die eine Reaktion auslösen. Typische Events in einer Android-Oberfläche sind:

- Klick auf Button oder Listeneintrag
- Textänderung in einem Eingabefeld
- Auswahl in einem Menü
- Scrollen, Wischen, lange Berührung
- Bestätigen oder Abbrechen

In Compose werden Events meistens über **Callback-Funktionen** behandelt. Das bedeutet:

- Die Oberfläche bietet eine "Stelle", an der ein Event gemeldet wird.
- Die Logik entscheidet, was passieren soll (zum Beispiel Navigation, Speichern, Validierung).

So bleibt die Oberfläche übersichtlich: Oberfläche zeigt an und meldet Events, Logik verarbeitet sie.

**Beispiel (Event als Callback):**

```kotlin
  @Composable
  fun SaveButton(onSaveClick: () -> Unit) {
    Button(onClick = onSaveClick) {
      Text("Speichern")
    }
  }
```

---

## State
**State** (Zustand) beschreibt, *was die Oberfläche gerade darstellen soll*. Beispiele:

- aktueller Text in einem Eingabefeld
- ob ein Dialog offen ist
- welche Elemente in einer Liste angezeigt werden
- ob ein Ladeindikator sichtbar ist
- ob ein Fehler angezeigt wird

In Compose ist der Zusammenhang zentral:

- **State ändert sich** -> Oberfläche passt sich an -> betroffene Teile werden neu aufgebaut

Damit wird die Oberfläche automatisch konsistent. Es muss nicht an vielen Stellen "manuell aktualisiert" werden.

**Beispiel (lokaler State mit `remember`):**

```kotlin
  @Composable
  fun Counter() {
    var count by remember { mutableStateOf(0) }

    Button(onClick = { count++ }) {
      Text(text = "Count: $count")
    }
  }
```

---

## Unidirektionaler Datenfluss
Ein wichtiges Prinzip beim GUI-Aufbau ist der **unidirektionale Datenfluss**:

- **State fließt nach unten** in die Oberfläche (Daten werden angezeigt)
- **Events fließen nach oben** zur Logik (Benutzeraktion wird gemeldet)

### Warum das wichtig ist
Dieses Prinzip verhindert typische Probleme:
- Zustand wird nicht "an mehreren Stellen" gleichzeitig verändert
- Logik ist nicht verstreut zwischen vielen UI-Elementen
- Oberfläche bleibt nachvollziehbar und einfacher zu testen

---

## Wo wird State gespeichert? (lokal oder zentral)
Je nach Größe und Bedeutung gibt es verschiedene Orte für State:

### Lokaler State (nur für ein UI-Detail)
Beispiele:
- ein Passwortfeld zeigt den Text kurz an oder versteckt ihn
- ein kleines Dropdown ist geöffnet oder geschlossen

Das ist sinnvoll, wenn der Zustand nur diesen kleinen Bereich betrifft.

### Zentraler State (Screen- oder App-Logik)
Beispiele:
- geladene Daten aus dem Backend
- Validierung von Eingaben
- Login-Status
- Ergebnisse, Filter, Sortierung

Dieser Zustand gehört meistens in eine zentrale Logik-Schicht (zum Beispiel ein ViewModel). Dadurch zeigt die Oberfläche nur das an, was sie bekommt.

---

## State Hoisting: Zustand bewusst nach oben ziehen
**State Hoisting** bedeutet: Zustand wird nicht im tiefen UI-Baustein versteckt, sondern in eine höhere Ebene verlagert.

Beispiel-Idee:
- Ein Eingabefeld zeigt Text an.
- Der Text wird aber nicht im Eingabefeld selbst dauerhaft gespeichert,
  sondern kommt von "oben" und Änderungen werden nach "oben" gemeldet.

Vorteile:
- mehrere Komponenten können denselben Zustand verwenden
- Oberfläche wird wiederverwendbar
- die Logik bleibt an einer Stelle

**Beispiel (State Hoisting mit TextField):**

```kotlin
  @Composable
  fun NameField(value: String, onValueChange: (String) -> Unit) {
    TextField(value = value, onValueChange = onValueChange)
  }

  @Composable
  fun NameScreen() {
    var name by remember { mutableStateOf("") }
    NameField(value = name, onValueChange = { name = it })
  }
```

---

## Typischer Ablauf in einer Compose-Oberfläche 

1. **Start**: Screen bekommt State (zum Beispiel Liste ist leer, Laden ist aktiv).
2. **Anzeige**: Oberfläche zeigt den State (Ladeindikator oder Liste).
3. **Event**: Benutzer klickt oder tippt.
4. **Weitergabe**: Oberfläche meldet das Event nach oben (Callback).
5. **Logik reagiert**: Daten werden geändert, gespeichert oder neu geladen.
6. **State aktualisiert sich**: neuer Zustand wird bereitgestellt.
7. **Compose aktualisiert UI**: Oberfläche passt sich automatisch an.

---

## Häufige State-Themen beim Aufbau einer Oberfläche
In einer Ausarbeitung sind diese Punkte oft besonders relevant:

- **Laden und Fehler anzeigen**:  
  State unterscheidet zwischen "lädt", "erfolgreich", "Fehler"
- **Formulare**:  
  Eingaben, Validierungsfehler, Button aktiv oder deaktiviert
- **Listen**:  
  leere Liste, gefüllte Liste, gefilterte Liste, Auswahl eines Eintrags
- **Navigation**:  
  Event löst Navigation aus, State bestimmt Inhalte des nächsten Screens

---

## Ergebnis: Was diese beiden Punkte für den GUI-Aufbau erklären
- **Composables, Layout, Modifiers** erklären, wie die Oberfläche strukturiert und gestaltet wird.
- **Events und State** erklären, wie die Oberfläche reagiert und warum sie in Compose automatisch korrekt bleibt.

Damit ist klar beschrieben, wie eine grafische Benutzeroberfläche in Jetpack Compose aufgebaut wird:  
**Struktur durch Layouts, Gestaltung durch Modifiers, Inhalt durch State, Interaktion durch Events.**
