# Navigation in Jetpack Compose

## Grundkonzept

Navigation ist ein System zum Wechseln zwischen verschiedenen Screens einer App.

**Zweck:**
- Benutzer zwischen verschiedenen Screens navigieren lassen (z.B. von Login zu Home)
- Zurück-Navigation automatisch verwalten (Back-Stack)
- Daten zwischen Screens übergeben
- Deep Links ermöglichen (externe Links zu bestimmten Screens)

**Was die Navigation Component bietet:**
- Back-Stack-Verwaltung (Zurück-Navigation)
- Deep Linking (Links zu bestimmten Screens)
- Typsicherheit
- ViewModel-Unterstützung

**Benötigte Library:**
```
androidx.navigation:navigation-compose
```

---

## Zentrale Komponenten

### NavController
- Steuert Navigation
- Verwaltet Back Stack
- Stateful

```kotlin
val navController = rememberNavController()
```

### NavHost
- UI-Container für den aktuellen Screen
- Verbindet NavController mit dem Navigation Graph

```kotlin
NavHost(
    navController = navController,
    startDestination = "home"
) {
    composable("home") { HomeScreen() }
    composable("details") { DetailsScreen() }
}
```

### Navigation Graph
- Struktur aller Screens
- Definiert Navigations-Beziehungen
- Kann verschachtelte Graphs enthalten

### Routes
- Eindeutige ID für einen Screen
- Kann Argumente enthalten

Beispiel:
```
"profile/{userId}"
```

---

## Navigation Operationen

**Navigieren:**

```kotlin
navController.navigate("profile/user123")
```

Type-safe (Kotlin Serialization):
```kotlin
navController.navigate(Profile(id = "user123"))
```

**Zurück:**

```kotlin
navController.popBackStack()
```

**Back Stack leeren:**

```kotlin
navController.navigate("main") {
    popUpTo("login") { inclusive = true }
}
```

---

## Argumente übergeben

```kotlin
composable(
    route = "profile/{userId}",
    arguments = listOf(
        navArgument("userId") { type = NavType.StringType }
    )
)
```

**Wichtig:**
- Keine komplexen Objekte übergeben
- Nur ID übergeben
- Daten aus zentraler Quelle laden (Single Source of Truth)

---

## SavedStateHandle

Argumente im ViewModel abrufen:

```kotlin
val userId = savedStateHandle["userId"]
```

Ermöglicht:
- Zugriff auf Navigationsargumente
- Rückgabe von Ergebnissen beim Zurücknavigieren

---

## Best Practices

- Navigation von UI entkoppeln
- NavController nicht direkt in Screens verwenden
- Callbacks übergeben
- startDestination muss feste Route sein
- Argumente minimal halten
- Typensicherheit nutzen

# ViewModel

## Grundkonzept

ViewModel ist eine Klasse, die UI-Daten und Logik für einen Screen verwaltet.

**Zweck:**
- **Daten bleiben bei Rotation erhalten**: Ohne ViewModel gehen alle Daten verloren, wenn das Handy gedreht wird
- **Logik vom UI trennen**: Business-Logik nicht in Activity/Composable -> bessere Testbarkeit
- **Daten-Cache**: API-Aufrufe müssen nicht bei jeder Rotation wiederholt werden
- **Zentrale Datenverwaltung**: Alle Screen-Daten an einem Ort

**Eigenschaften:**
- Speichert den Status eines Screens
- Liefert UI-Status
- Kapselt Business-Logik
- **Überlebt Änderungen der Konfiguration** (z.B. Rotation)
- Verhindert Datenverlust

---

## Lebenszyklus

ViewModel ist an einen **Owner** (Besitzer) gebunden:
- **Activity** -> ViewModel bleibt bis Activity beendet ist
- **Fragment** -> bis Fragment entfernt ist
- **NavBackStackEntry** -> bis der Screen aus dem Backstack entfernt ist

**Wichtig:** Rotation zerstört die Activity, aber nicht das ViewModel.

---

## Speicherlecks vermeiden

**ViewModel darf NICHT enthalten:**
- Verweise auf UI/Lifecycle-Objekte (z.B. `Context`, `Resources`, Views)
- UI-Details
- Weitergabe an andere Klassen

---

## ViewModel erstellen

**Activity (Kotlin):**
```kotlin
import androidx.activity.viewModels

val viewModel: DiceRollViewModel by viewModels()
```

**Compose:**
```kotlin
import androidx.lifecycle.viewmodel.compose.viewModel

@Composable
fun Screen(viewModel: DiceRollViewModel = viewModel()) { }
```

---

## UI-State Pattern

```kotlin
data class DiceUiState(
    val firstDieValue: Int? = null,
    val secondDieValue: Int? = null,
    val numberOfRolls: Int = 0,
)

class DiceRollViewModel : ViewModel() {
    private val _uiState = MutableStateFlow(DiceUiState())
    val uiState: StateFlow<DiceUiState> = _uiState.asStateFlow()

    fun rollDice() {
        _uiState.update { s ->
            s.copy(
                firstDieValue = Random.nextInt(1, 7),
                secondDieValue = Random.nextInt(1, 7),
                numberOfRolls = s.numberOfRolls + 1
            )
        }
    }
}
```

**Pattern:**
- Activity/Composable beobachtet nur `uiState`
- Ruft Funktionen auf (z.B. `rollDice()`)
- Daten holen/speichern im ViewModel

---

## SavedStateHandle

Ermöglicht Speicherung des Zustands auch bei **Prozess-Neustart**.

**Unterschied:**
- ViewModel: überlebt Rotation
- SavedStateHandle: überlebt auch wenn Android den Prozess beendet

---

## Coroutines und viewModelScope

`viewModelScope` ist ein CoroutineScope, das automatisch endet, wenn ViewModel gelöscht wird.

Alternative für Tests:

```kotlin
override fun onCleared() {
    coroutineScope.cancel()
}
```

# Room Database

## Grundkonzept

Room ist eine Library für lokale Datenspeicherung auf dem Gerät. Es ist eine Vereinfachung von SQLite.

**Zweck:**
- **Daten lokal speichern**: Daten bleiben auch nach App-Neustart erhalten
- **Offline-Funktionalität**: App funktioniert ohne Internet
- **Cache für API-Daten**: Schnellerer Zugriff, weniger Netzwerk-Anfragen
- **Benutzerdaten speichern**: Einstellungen, Favoriten, etc.
- **Typsicherheit**: Fehler werden beim Kompilieren erkannt, nicht beim Ausführen

**Vorteile gegenüber SQLite direkt:**
- Typsichere Datenbankzugriffe
- Fehlerprüfung beim Kompilieren
- Weniger Wiederholcode
- Einfache Migrationen (Datenbank-Updates)
- Automatische UI-Updates mit Flow

**Aufbau - 3 Hauptkomponenten:**
- **Entity** (Tabelle) -> Die Datenstruktur
- **DAO** (Data Access Object) -> Die Zugriffsmethoden
- **RoomDatabase** -> Die Datenbank selbst

---

## Entity Definition

```kotlin
@Entity
data class User(
    @PrimaryKey val id: Int,
    val name: String
)
```

**Wichtig:**
- @PrimaryKey ist Pflicht
- Jede Entity = eine Tabelle

---

## DAO (Data Access Object)

```kotlin
@Dao
interface UserDao {

    @Query("SELECT * FROM user")
    fun getAll(): List<User>

    @Insert
    suspend fun insert(user: User)
}
```

**Annotationen:**
- `@Query`
- `@Insert`
- `@Update`
- `@Delete`

---

## Database Klasse

```kotlin
@Database(entities = [User::class], version = 1)
abstract class AppDatabase : RoomDatabase() {
    abstract fun userDao(): UserDao
}
```

- Annotiert mit `@Database`
- Erbt von `RoomDatabase`
- Enthält abstrakte DAO-Methoden

---

## Datenbank Instanz erstellen

```kotlin
val db = Room.databaseBuilder(
    context,
    AppDatabase::class.java,
    "app-db"
).build()
```

**Wichtig:**
- Nur eine Instanz verwenden (Singleton)
- Erstellung ist teuer

---

## Coroutines Integration

**Regel:** Room verbietet Datenbank-Zugriffe am Main Thread (UI-Thread).

**Einmalige Abfragen (suspend):**

```kotlin
@Insert
suspend fun insert(user: User)

@Query("SELECT * FROM user WHERE id = :id")
suspend fun getById(id: Int): User
```

---

## Beobachtbare Abfragen mit Flow

```kotlin
@Query("SELECT * FROM user")
fun getAll(): Flow<List<User>>
```

**Eigenschaften:**
- Automatische Aktualisierung bei Tabellenänderung
- Ideal für Compose

---

## Verwendung in Compose

```kotlin
@Composable
fun UserScreen(vm: UserViewModel) {
    val users by vm.users.collectAsState(initial = emptyList())
}
```

Flow -> collectAsState -> UI reagiert automatisch.

---

## Migrationen

### Automatische Migration (bei einfachen Änderungen)

```kotlin
@Database(
    version = 2,
    entities = [User::class],
    autoMigrations = [AutoMigration(from = 1, to = 2)]
)
```

### Manuelle Migration (bei komplexen Änderungen)

```kotlin
val MIGRATION_1_2 = object : Migration(1, 2) {
    override fun migrate(db: SupportSQLiteDatabase) {
        db.execSQL("ALTER TABLE user ADD COLUMN age INTEGER")
    }
}
```

Registrierung:

```kotlin
Room.databaseBuilder(...)
    .addMigrations(MIGRATION_1_2)
    .build()
```

### Fallback 

**VORSICHT**: Gefährlich, löscht die komplette Datenbank und erstellt sie neu.

```kotlin
fallbackToDestructiveMigration()
```

---

# Zusammenfassung

**Navigation:**
- NavController + NavHost + Routes
- Argumente minimal halten (nur IDs)
- Navigation vom UI trennen

**ViewModel:**
- Überlebt Rotation und Konfigurationsänderungen
- Keine UI-Verweise (sonst Speicherlecks)
- UI-State Pattern mit StateFlow verwenden

**Room:**
- Entity -> Tabelle
- DAO -> Zugriffsmethoden
- Kein Main-Thread-Zugriff (immer suspend oder Flow)
- Flow für automatische UI-Updates
- Singleton-Muster verwenden (nur eine Instanz)
